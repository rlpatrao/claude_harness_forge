## Background — what triggered v3.1

User returned after a long absence (last active 2026-05-20). After surveying state, they asked for two concrete changes plus exploratory work on two concepts:

1. **Scaffold's first question must offer a choice:** interactive Q&A *or* pre-existing artifacts (BRD.md, Architecture.md, AAC files).
2. **When architecture is not provided, architect must present a synthesized recommendation back to the user for approval before the autonomous loop starts building.**
3. *Exploratory:* evaluate **memory_os** patterns (MemGPT/Letta/Mem0/Zep — 3-tier core/recall/archival) for the harness.
4. *Exploratory:* evaluate Addy Osmani's **loop engineering** (https://addyosmani.com/blog/loop-engineering/) — five primitives: automations, worktrees, skills, plugins, sub-agents, plus state.

## 2. Increment v3.1.1 — Scaffold Q0: Q&A vs Artifacts

### Why

Today [`commands/scaffold.md`](../commands/scaffold.md) forces interactive Q&A through `/brd` → `/architect`. Users with pre-existing BRDs (org-standard, exported from Jira/Confluence/Linear) hit pointless interviews. A first-question branch turns this into a choice.

### What changes

Insert Q0 before existing Q1 ("What are you building?"):

```
Q0. How are you providing requirements and architecture for this project?
  A) Interactive Q&A — current flow: /brd interview → /architect interrogation
  B) I have a pre-created BRD and/or Architecture doc — import them
  C) Hybrid: I have a BRD but no architecture — interview for architecture only
```

**Branch A:** unchanged — current scaffold flow.

**Branch B:** scaffold prompts for paths to:
- `requirements.md` or `BRD.md` → staged into `specs/brd/app_spec.md`
- `architecture.md` or `design.md` → staged into `specs/design/architecture.md`
- (optional) AAC files in `*.dsl` / `*.puml` / `*.mmd` — deferred to v3.1.3; v3.1.1 supports markdown only
- Marks `specs/brd/.imported` and `specs/design/.imported` sentinels so downstream agents skip interview

**Branch C:** import BRD, run architect interview only (skips `/brd`).

### Files to create

| Path | Purpose |
|---|---|
| `skills/scaffold-import/SKILL.md` | New skill: validates imported files, stages them, writes sentinels |
| `skills/scaffold-import/templates/import-checklist.md` | What we accept and what we require per branch |

### Files to update

| Path | Change |
|---|---|
| `commands/scaffold.md` | Insert Q0 + three branches before existing Step 1 |
| `skills/brd/SKILL.md` | At entry: if `specs/brd/.imported` exists, skip interview, exit success |
| `skills/architect/SKILL.md` | At entry: if `specs/design/.imported` exists, set `--from-import` mode (v3.1.2 implements behavior) |
| `agents/brd-creator.md` | Add early-exit when `.imported` sentinel present |

### Verification (v3.1.1)

Three test scaffolds, each into a throwaway directory:

1. **Pure Q&A** (Branch A): no artifacts supplied → confirm `/brd` interview runs, `/architect` interrogation runs (unchanged behavior).
2. **Pure import** (Branch B): supply `brd.md` + `architecture.md` → confirm both stage into `specs/`, sentinels written, `/brd` and `/architect` skip interview when invoked.
3. **Hybrid** (Branch C): supply `brd.md` only → confirm `/brd` skip, `/architect` runs interview.

Artifact: `verification/v3.1.1-scaffold-import-mode.json` capturing the three test runs' file listings + skip-decisions.

## 3. Increment v3.1.2 — Architect synthesis mode + review loop + auto-handoff

### Why

Two related gaps:

1. When architecture is imported, the existing 11-round interrogation in [`skills/architect/SKILL.md`](../skills/architect/SKILL.md) is redundant and annoying.
2. When architecture is *not* imported, the current architect writes design artifacts piecewise across rounds. User asked for a single synthesized recommendation presented for explicit approval (approve / amend / restart) **before** the autonomous loop starts building.

### What changes

**Synthesis mode** (`--from-import`):
- Architect reads `specs/brd/app_spec.md` + `specs/design/architecture.md` + `learnings/stack-decisions/`.
- Skips Rounds 1–11.
- Synthesizes derived design artifacts (api-contracts, data-models, component-map, folder-structure, deployment) directly.
- Goes to review loop.

**Review loop** (runs after interview OR synthesis):
- Architect emits a single document: `specs/design/architecture-review-v{N}.md` containing the full proposed architecture, all stack decisions, the decision rationale, and explicit "alternatives considered."
- Presents to user with 3 options:
  - **Approve** → write final `specs/design/architecture.md`, emit handoff note, optionally trigger `/auto`.
  - **Amend** → user supplies inline feedback; architect produces `architecture-review-v{N+1}.md`. Budget = 3 amend cycles; on cycle 4, force restart.
  - **Restart** → drop the synthesized output, fall back to full 11-round interview.

**Auto-handoff:**
- On approve, [`hooks/session-start.js`](../hooks/session-start.js) detects `state/architecture-approved.flag` on next invocation and surfaces "Architecture approved at {timestamp}. Suggested next: `/auto`" in the SessionStart system reminder.
- We do NOT auto-invoke `/auto` silently — that's a destructive automation under the executing-actions-with-care principle. We surface and let the user confirm with one keystroke.

### Files to create

| Path | Purpose |
|---|---|
| `skills/architect/templates/architecture-review.md` | Single-doc review template (sections: Stack, Data, API, Deployment, Decisions, Alternatives, Risks) |
| `skills/architect/synthesis-mode.md` | Inline doc embedded into SKILL.md describing the `--from-import` decision tree |

### Files to update

| Path | Change |
|---|---|
| `skills/architect/SKILL.md` | Add Step 0 (mode detection: `--from-import` ⇒ synthesis; else interview). Add Step 5 (review loop). Existing Steps 1–4 unchanged. |
| `agents/architect.md` | Same mode-detection logic + review-loop instructions |
| `commands/architect.md` | Accept `--from-import` flag (set automatically by scaffold Branch B/C); document `--restart` as override |
| `hooks/session-start.js` | Detect `state/architecture-approved.flag` on session start; emit suggestion to invoke `/auto` |
| `state/` | New file (created at runtime by architect, not at scaffold): `architecture-approved.flag` with timestamp + version |

### Verification (v3.1.2)

1. **Synthesis-mode**: invoke `/architect --from-import` against a project with imported artifacts → confirm no interview rounds asked, design artifacts produced, review doc emitted.
2. **Review-approve**: approve `architecture-review-v1.md` → confirm final `architecture.md` written, `architecture-approved.flag` set, next SessionStart shows `/auto` suggestion.
3. **Review-amend**: amend twice, approve on v3 → confirm v1/v2/v3 docs exist, v3 promoted to final, flag set.
4. **Review-restart**: restart on v1 → confirm fallback to interview, v1 doc archived to `specs/design/amendments/`.
5. **Amend budget**: amend 3× without approving → confirm cycle 4 forces restart with informative message.

Artifact: `verification/v3.1.2-architect-review-loop.json` capturing each of the 5 test runs' state transitions.

## 4. Future increments — sketched, not committed

### v3.1.3 — Harness registry (borrow from claude_harness_eng_v5)

Create `HARNESS.md` + `harness-manifest.json` at forge root modeling every agent/skill/hook/gate as a row in the `{maintainability, architecture, behaviour, traceability} × {planning, session, commit, integration, drift}` matrix. Each row has `axis`, `cadence`, `type` (computational/inferential/hybrid), `scope`, `wired_at` path, `status`. Add `scripts/validate-harness-manifest.js` to CI. Add `scripts/harness-coverage.js` to compute per-axis coverage against the code graph (v3.1.9). This is the framing artifact that makes forge gaps enumerable.

Source: [`cwijayasundara/claude_harness_eng_v5/HARNESS.md`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/HARNESS.md), [`harness-manifest.json`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/harness-manifest.json).

### v3.1.4 — Worktree-by-default

Agents currently edit main checkout. Move to:
- `settings.json: worktree.bgIsolation: "worktree"` (already in CC schema)
- Agent spawn paths use `isolation: "worktree"` param
- Tests for parallel agents not colliding on shared files

Addy Osmani's "**Worktrees prevent parallel agents from colliding**" — first-class gap.

### v3.1.5 — Defensive triad (borrow from claude_harness_eng_v5)

Three small, defensive borrows that close real holes:
- **Real git hooks** — `templates/git-hooks/pre-commit` + `prepare-commit-msg` installed by scaffold Step 8. Fires on `git commit --amend` and shell git where Node PreCommit hooks miss. Adds staged-file layer scan, tsc noEmit, pytest coverage ratchet against baseline.
- **`hooks/pre-bash-gate.js`** — extracts write targets from bash (`>`, `>>`, `tee`, `sed -i`, `dd`, `cp`, `mv`) and re-applies scope + env protection. Closes the `echo secret > .env` bypass.
- **`hooks/concurrency-gate.js`** — caps concurrent `Task`/`Agent` subagents at 18 (configurable). Decrements on `SubagentStop`. TTL-prunes leaked counts. Prevents runaway fan-out on parallel `generator` teammate spawns.

Each is <100 lines. Sources: [`git-hooks/pre-commit`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/.claude/git-hooks/pre-commit), [`pre-bash-gate.js`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/.claude/hooks/pre-bash-gate.js), [`concurrency-gate.js`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/.claude/hooks/concurrency-gate.js).

### v3.1.6 — Token Governor MVP (borrow from claude_harness_eng_v5)

Directly attacks token waste on large repos. Ship:
- `commands/context.md` — `/context "<question>"` returns bounded `file:line` citations before broad Reads.
- `scripts/context-pack.js`, `scripts/context-retrieve.js`, `scripts/search-compact.js`, `scripts/run-compact.js` — CCR (compress-cache-retrieve) pipeline; raw command/search output stored under `state/context-cache/<hash>/` and re-served as a failure-first digest.
- `hooks/token-advisor.js` — PreToolUse(Read|Bash) advisory hook that warns on avoidable broad reads.

Complements the `context-budget` and `context-engineering` skills, which currently document the problem without providing tooling.

Source: [`docs/token-governor.md`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/docs/token-governor.md) + `.claude/scripts/context-*.js`.

### v3.1.7 — Triage inbox

New skill `skills/triage/SKILL.md` + surface `specs/triage/inbox.md`. Discovery automations (v3.1.8) write here; humans skim before deciding. Distinct from `feature_list.json` (which is the committed work queue).

### v3.1.8 — Scheduled automations

Wrap the existing `/loop` slash command + cron + GH Actions templates. `recipes/example-morning-triage.yaml`. The Stop-hook re-entry (Ralph Loop) covers within-session re-entry; this covers across-session scheduling.

### v3.1.9 — Living code-graph + graph-refresh + codebase-explorer (borrow from claude_harness_eng_v5)

- **`skills/code-map/SKILL.md`** — AST-first indexer producing `state/code-graph.json`, `state/symbol-map.md`, `wiki/WIKI.md`.
- **`hooks/graph-refresh.js`** — Stop hook drains a dirty-file jsonl (fed by `verify-on-save`-analog on Write/Edit) and re-runs indexer with `--files` mode. Graph is always fresh for the next turn without per-edit cost.
- **`agents/codebase-explorer.md`** — read-only exploration agent with LSP tool grant (pyright, ts-language-server, gopls, jdtls, omnisharp, rust-analyzer). Grounds every "where does X live" query in `file:line` citations.

Powers seam-finding, modularity review, drift detection, verification-matrix, memory-OS archival tier. Unlocks brownfield.

Source: [`.claude/skills/code-map/`](https://github.com/cwijayasundara/claude_harness_eng_v5/tree/main/.claude/skills/code-map), [`.claude/hooks/graph-refresh.js`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/.claude/hooks/graph-refresh.js), [`.claude/agents/codebase-explorer.md`](https://github.com/cwijayasundara/claude_harness_eng_v5/blob/main/.claude/agents/codebase-explorer.md).

### v3.1.10 — AAC parsers

Three parsers (Structurizr DSL, PlantUML C4, Mermaid C4), common IR, then reuses v3.1.2 synthesis pipeline. Edge cases (Structurizr DSL macros, PlantUML `!include`, Mermaid theme directives) are why this is a separate increment.

### v3.1.11 — Memory OS, filesystem tier

Three Claude tools backed by structured files (no vector DB yet):

| Tool | Backing | Behavior |
|------|---------|----------|
| `core_memory_read` / `core_memory_write` | `state/memory/core-blocks/*.md` | Block-structured always-in-context |
| `recall_search` | `state/memory/sessions/*.jsonl` + ripgrep | Search prior session transcripts (post-compaction summaries kept) |
| `archival_search` | `state/memory/archival/*.md` + ripgrep + BM25 | Long-term notes the agent writes itself; searchable |

Filesystem-first because (a) inspectable in git, (b) no vendor lock, (c) defers schema decisions until usage patterns clarify. **Prereq:** v3.1.9 code-graph feeds the archival index.

### v3.1.12 — Memory OS, vector backend

Optional swap of `archival_search` backend to Mem0 (community-favored, 48k★, $24M Series A) or Letta (MemGPT successor, full memory-runtime). MCP server, not in-process. Decision deferred until v3.1.11 has been in use for ≥2 dogfood projects.

### v3.1.13 — Dogfood end-to-end

Target: [`test-projects/salary-dashboard/`](../test-projects/salary-dashboard/), scaffolded 2026-05-20 with partial BRD captured in conversation memory. Use it to verify the upgraded scaffold (import path via the partial BRD), the architect review loop, the harness registry, the token governor, and the code-graph.
