# v3.5 — Port to GitHub Copilot: effort analysis

**Status:** analysis only — no code shipped. Sits alongside `v3.4-headless-dogfood.md` as a design candidate for the next increment.
**Date:** 2026-07-22.
**Verdict:** ~1.5–2.5 weeks of work. Cheaper than the Codex CLI port (which was 2-4 weeks) because Copilot CLI's hook and agent architecture aligns closely with Claude Code's.

---

## TL;DR

GitHub Copilot CLI (as of mid-2026) natively supports:

- **Hooks with 14 events** — a superset of our 7. `sessionStart` / `preToolUse` / `postToolUse` / `preCompact` / `agentStop` (=our Stop) / `subagentStart` + `subagentStop` all present.
- **Exit-code semantics identical to ours** — `0` = success, `2` = block on `preToolUse` / `permissionRequest`.
- **`decision: "block"` + `reason`** in stdout JSON — same shape.
- **MCP servers** with auto-approve at server + tool level.
- **Custom subagents** as `.agent.md` files in `.github/agents/`.
- **AGENTS.md, CLAUDE.md, GEMINI.md**, and `.github/copilot-instructions.md` all read as custom instructions — **our existing `CLAUDE.md` ports for free**.
- **Custom slash commands** (analog to `/scaffold`, `/auto`, etc.).

Every architectural primitive the forge relies on is present in Copilot. The port is a translation, not a rewrite.

## 4. The one real risk — LLM billing / routing

Copilot CLI runs models through **Copilot's own inference**, billed through Copilot Business/Enterprise. **You cannot BYOK.** Consequences:

- Our `workflows.yaml` routing (Claude Opus for critic, Kimi K3 for coding-agent, GPT-5 fallback) becomes moot on Copilot. Copilot picks the model per `.agent.md`'s `model:` field from Copilot's roster.
- Copilot's roster in 2026 includes GPT-5, GPT-5-codex, Claude Sonnet, Claude Opus, Gemini 2.5 Pro — all available. So the *models* we want are there; the routing config just moves from us to them.
- Our failover chain (Anthropic → OpenAI on 429) is handled internally by Copilot, not through our config.

**Net:** you lose per-workflow BYO-LLM control; you gain per-user Copilot pooled billing (attractive for teams already paying Copilot Business).

## 5. The one real gotcha — cloud agent constraints

If you use Copilot's **cloud coding agent** (autonomous PRs from GitHub Actions, not the local CLI), extra constraints apply:

- `/workspace` is the only writable dir
- Outbound network restricted (no arbitrary curl)
- Only `bash` in hook config (not PowerShell)
- No user Y/N prompts — everything must be non-interactive

**Our v3.4 headless work already fits this exactly.** `--branch B --brd --arch --auto-approve` + `AUTO_ADVANCE_ON_ARCHITECTURE_APPROVED=1` is precisely what the cloud agent wants. This is the one place where v3.4 pays off structurally: the same design that made the forge testable under CI makes it Copilot-cloud-agent-ready without extra work.

## 6. What needs work — the translation layer

### 6a. Output adapter (~0.5 day)

Write `hooks/lib/output.js`:

```js
// Dual-format hook output emitter.
// Emits Claude Code shape by default; Copilot flat shape when COPILOT=1.
function emit(event, obj) {
  const isCopilot = process.env.COPILOT === '1';
  if (isCopilot) {
    // Flat shape: { additionalContext, decision, reason, permissionDecision, ... }
    process.stdout.write(JSON.stringify(obj));
  } else {
    // Claude Code shape: { hookSpecificOutput: { hookEventName, additionalContext }, decision, reason, ... }
    const out = { ...obj };
    if (obj.additionalContext !== undefined) {
      out.hookSpecificOutput = { hookEventName: event, additionalContext: obj.additionalContext };
      delete out.additionalContext;
    }
    process.stdout.write(JSON.stringify(out));
  }
}
module.exports = { emit };
```

Update our ~10 hooks that emit `hookSpecificOutput.additionalContext` to call `emit()` instead. Backward compatible with Claude Code — no behavior change unless `COPILOT=1`.

### 6b. Hook config exporter (~0.5 day)

`scripts/export-hooks-to-copilot.js`: reads our `settings.json` `hooks` block, writes `.github/hooks/*.json` in Copilot's per-file format. Idempotent — re-run to update.

### 6c. Agent exporter (~1-2 days)

`scripts/export-agents-to-copilot.js`: iterates `agents/*.md`, copies to `.github/agents/<name>.agent.md` with frontmatter mapping:
- `model_preference: opus` → `model: claude-opus`
- `tools: [Read, Write, Bash]` → `tools: read, write, bash` (Copilot format)
- `{{model:<workflow>}}` placeholders → resolve from `config/workflows.yaml`

### 6d. Commands exporter (~1 day)

Same shape as 6c but for `commands/*.md`.

### 6e. MCP config move (~0.5 day)

Move `.claude-plugin/plugin.json` MCP declarations to Copilot's MCP config (location TBD; likely `.github/copilot-mcp.json` or per-agent frontmatter).

### 6f. AGENTS.md landing page (~0.5 day)

Author a top-level `AGENTS.md` that includes CLAUDE.md by reference and adds Copilot-specific guidance (auto-approve setup, `.github/agents/` install path).

## 7. Dogfood on Copilot (~3–5 days)

The real risk lives here.

1. Run `scripts/export-*.js` to produce `.github/{hooks,agents}/` from forge sources
2. `git commit` the exports
3. Start Copilot CLI on `test-projects/salary-dashboard/` — either interactive (`copilot`) or headless via `COPILOT=1 AUTO_ADVANCE_ON_ARCHITECTURE_APPROVED=1 copilot --print /auto`
4. Watch `state/fire-log.jsonl` — hooks should fire with the same names as under Claude Code
5. Compare parity: same feature ends `passes:true`? Same verification artifact? Same commit sequence?

Expected first-run issues:
- Subagent tool-call format differences (Copilot may use different tool names)
- MCP server startup edge cases (Copilot's MCP client may enforce stricter timeouts)
- Permission grants (Copilot's permission model isn't identical to ours)
- Slash command discovery (Copilot may look for commands in a different path)

Fix each, re-run, converge.

## 8. Bottom-line recommendation

**Do it if:**
- Your team is on Copilot Business/Enterprise (billing is pooled through Copilot)
- You want the forge available as a `.github/agents/` install rather than requiring `claude --plugin-dir`
- You want the cloud-agent CI path (`copilot /agent` in a GitHub Actions workflow → autonomous PR)

**Don't do it if:**
- You need BYO-LLM routing (Kimi K3 direct, local Qwen, custom failover) — Copilot boxes you in
- You want tree-structured sessions with per-branch fork history — Copilot's session model doesn't match our v3.0 §4.5 tree-sessions
- You're the only user of the forge and Claude Code works fine

**Effort ballpark:** budget **2 weeks of focused work** for a working port, **3 weeks** if you want fire-log parity and cloud-agent CI runs green.
