# BRD v3.4 — Headless scaffold + dogfood

**Status:** landed. Both halves — flags + dogfood setup — shipped in a single session (2026-07-14).
**Motivation:** v3.1.1 (scaffold-import Branch B) + v3.1.2 (architect synthesis mode) already made most of scaffold non-interactive when BRD + Architecture artifacts are provided upfront. Three touchpoints still required a human keystroke; v3.4 removes them so the dogfood can run under CI or an SDK driver with zero prompts.

---

## What v3.4 changes

### B — CLI flags + env var (headless mode)

| Change | Where | Effect |
|---|---|---|
| `--branch A\|B\|C` | `commands/scaffold.md` Step -1 | Sets Q0 answer. Skips the interactive Q0 prompt. |
| `--brd <path>` | `commands/scaffold.md` Step 0 Branch B/C | Path to BRD source; scaffold-import stages it into `specs/brd/app_spec.md`. |
| `--arch <path>` | `commands/scaffold.md` Step 0 Branch B | Path to Architecture source (Markdown, `.dsl`, `.puml`, or `.mmd`). AAC files run through v3.1.10 parsers. |
| `--name <name>` | `commands/scaffold.md` Step 1 Q1 | Skips "What are you building?" prompt. |
| `--type saas\|enterprise\|api-only` | `commands/scaffold.md` Step 1 Q2 | Skips project-type prompt. |
| `--plugins minimal\|full\|none` | `commands/scaffold.md` Step 1 Q3 | Skips plugin picker. |
| `--yes` | any Y/N confirmation | Accepts default (e.g. scaffold-import overwrite prompts). |
| `--auto-approve` | `commands/architect.md`, `skills/architect/SKILL.md`, `agents/architect.md` | In synthesis mode only: skips the Step 6 A/M/R review-loop prompt and auto-approves v1. Refused in interview mode. |
| `AUTO_ADVANCE_ON_ARCHITECTURE_APPROVED=1` | `hooks/session-start.js` + `agents/coding-agent.md` | When the flag file exists AND the env var is set, SessionStart emits an IMPERATIVE ("Invoke `/auto` as your first action — do NOT prompt the human"). Coding-agent step 3a acts on it. |

Every flag is opt-in per-question. Omit a flag and the corresponding prompt runs as before. A fully-headless invocation:

```
/scaffold --branch B --brd path/to/BRD.md --arch path/to/architecture.dsl \
          --name salary-dashboard --type saas --plugins minimal --yes
```

...asks zero questions.

Followed by architect in synthesis + auto-approve:

```
/architect --from-import --auto-approve
```

...produces `state/architecture-approved.flag` with no prompts.

Then, in a subsequent (or the same) session with the env var set:

```
AUTO_ADVANCE_ON_ARCHITECTURE_APPROVED=1 claude
```

...triggers `/auto` on session start.

### A — Dogfood setup script + salary-dashboard fixtures

| File | Purpose |
|---|---|
| `templates/dogfood-fixtures/salary-dashboard/BRD.md` | Full BRD (5 dimensions, 5 ACs) for the salary dashboard target. Sourced from the paused 2026-05-20 interview + a self-contained rewrite. |
| `templates/dogfood-fixtures/salary-dashboard/architecture.dsl` | Structurizr DSL for the architecture (system + 5 containers: Web/API/DB/LLM-Proxy/ETL + 3 external systems). |
| `scripts/dogfood-setup.sh` | The setup script. Reads a fixture directory, stages every artifact the forge expects, writes `state/architecture-approved.flag`, seeds `feature_list.json` + `state/{compiled-rules,learned-rules}.md`, and commits an initial state. Leaves the target ready for `/auto`. |

`dogfood-setup.sh` does NOT depend on B's flags — it writes the artifacts directly, which is even more headless than invoking `/scaffold` with flags. Both paths reach the same end state; the script is faster for CI use, the flags are cleaner for interactive-first use.

## How to use

### For a v3.4-style dogfood run (recommended for CI, batch runs, dogfooding the harness itself)

```bash
# Setup — zero prompts, ~100ms
./scripts/dogfood-setup.sh --target ./test-projects/salary-dashboard --fixture salary-dashboard

# Start an interactive session that auto-advances
cd ./test-projects/salary-dashboard
AUTO_ADVANCE_ON_ARCHITECTURE_APPROVED=1 claude
```

The session starts, sees the imperative banner, invokes `/auto`, and runs the full pipeline. From this point on the driver session is still an interactive Claude Code session (the SDK needs one to spawn Task/Agent subagents, run Playwright/Puppeteer MCP, drive tool calls) — but every human decision has been made upfront via the fixture + the auto-approve semantics.

### For an interactive dogfood run (recommended when authoring a new fixture)

Use plain `/scaffold` and choose branch B interactively. Pass the flags you want to skip; leave the rest interactive.

### To author a new fixture

```
templates/dogfood-fixtures/<name>/
  BRD.md           # required — the source-of-truth requirements doc
  architecture.md  # OR
  architecture.dsl # OR
  architecture.puml# OR
  architecture.mmd # any ONE architecture artifact
```

Then:

```bash
./scripts/dogfood-setup.sh --target ./somewhere --fixture <name>
```

The parser is auto-detected from the extension. Structurizr DSL and Mermaid C4 are the least ambiguous formats; PlantUML C4 and Markdown work but the parsers are more conservative.

## Why we split B and A

- **B alone** (flags) doesn't need A — you can pass the flags to `/scaffold` and drive from there. Useful when someone else authored the BRD + Architecture in-repo.
- **A alone** (setup script) doesn't need B — the script writes files directly, bypassing scaffold. Useful when the fixture lives outside the target repo.
- **Together** — CI can call `dogfood-setup.sh --fixture salary-dashboard` and never touch scaffold. Interactive users can call `/scaffold --branch B --brd fixture/BRD.md --arch fixture/architecture.dsl` and get the same end state.

The two implementations converge on the same on-disk invariants (`specs/brd/.imported`, `specs/design/.imported`, `state/architecture-approved.flag`, `feature_list.json` seed).

---

## What still needs a human

Even with B + A, the following are still under the human's control:

1. **Authoring the fixture.** The BRD + Architecture DSL are the human contract with the forge. There's no plan to synthesize these — that would defeat the point.
2. **Reviewing the initial commit.** `dogfood-setup.sh` makes one commit; the human reviews it (via `git log -1 -p`) before starting the auto session.
3. **Deciding when to promote tentative → confirmed rules.** v3.3 `/rules promote-tentative` doesn't auto-promote; that's by design.
4. **Interactive dogfood driver session.** The SDK needs an interactive Claude Code session to run — spawns Task/Agent subagents, runs MCP tools, drives the build. What v3.4 removes is every prompt within that session that could be answered upfront.

## Deferred (not in v3.4)

- **`/scaffold` flag parser as a real parser** — currently the command's markdown instructions ask the model to parse `$ARGUMENTS`. A shell-side parser (like `getopts`) inside a script would be more robust. Deferred; the current prose approach works for all flag combinations we've tested.
- **Fixture library** — v3.4 ships one fixture (salary-dashboard). Adding fixtures is additive; no forge changes needed.
- **CI job that runs dogfood-setup + drives /auto** — needs an SDK-based driver (headless Claude Code). Left for a follow-on increment once we know the shape of that driver.
- **Batch-mode dogfood output aggregation** — running multiple fixtures + summarizing pass/fail across them. Deferred until we have more fixtures.
