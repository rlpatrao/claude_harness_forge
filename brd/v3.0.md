# claude_harness_forge — BRD v3.0

## 0. What changed from v2.0

v2.0 built a 9-phase pipeline with GAN-inspired generator/critic, a Karpathy-style eval ratchet, 10 agents, 23 skills, 14 hooks, MCP as universal tool protocol, and a multi-cloud adapter (ACA Dynamic Sessions, Durable Functions, supervised HITL gates, permission classifier).

v3.0 is not a rewrite. It is a **retrofit and re-architecture** driven by three documents and selective borrowing from five repos:

1. **Anthropic, "Effective harnesses for long-running agents" (Nov 2025)** — initializer/coding agent split, feature_list.json as the contract, claude-progress.txt + git as the cross-session bridge, mandatory browser-automation end-to-end testing.
1. **OPENDEV technical report (arXiv 2603.05344, Mar 2026)** — compound AI system, per-workflow LLM routing, dual-agent Plan/Normal modes via subagent delegation (not state machines), Extended ReAct (thinking + critique + action), five-layer defense-in-depth, event-driven system reminders, adaptive context compaction, three-tier skills hierarchy.
1. **Pi-mono (Mario Zechner)** — radical minimalism, tree-structured sessions, cross-provider mid-session handoffs, per-session cost tracking, message queuing while agent executes.

Plus selectively borrowed: Forge spec-gap backpropagation, The Factory monotonic-improvement guards, ECC continuous-learning v2 instinct extraction, Goose YAML recipes, Ralph Loop exit-interception, Stripe/Google budget-footer injection.

The architectural verdict from v2.0 review still holds: **don't ditch, retrofit**. v3.0 closes eight specific gaps without disturbing the multi-cloud adapter or the existing Durable Functions topology.

**Reuse mandate.** v3.0 explicitly leverages existing public implementations rather than rewriting from scratch. Every component below lists the upstream repo, the file/module to harvest, the license, and the integration mode (vendor, fork, depend, or read-and-reimplement). The token cost of generating any component from scratch must be weighed against the cost of vendoring it from a working reference. **Default to vendoring; reimplement only when the license or architecture forbids it.** See §10 for the full repo-harvest map.

## 1. Design principles (v3.0)

Five principles govern every v3.0 decision. They override prior conventions where they conflict.

1. **Separation of concerns.** Model selection, context management, safety enforcement, tool dispatch, persistence — each is independently configurable and replaceable. OPENDEV principle, adopted verbatim.
1. **Progressive degradation.** Graceful function as resources exhaust (tokens, iterations, network). No cliff failures.
1. **Transparency over magic.** Every tool call, safety veto, compaction, memory write, ratchet decision, and model route is observable and overridable.
1. **Goal-driven, not instruction-driven.** Karpathy framing. Tasks intake as declarative success criteria with verification loops; never as imperatives.
1. **Compound system over monolithic agent.** Workflows bind to models, not vice versa. Per-workflow LLM routing is the default.

## 2. Architecture: four layers + harness

Inherited four-layer organization from OPENDEV, mapped onto the existing multi-cloud adapter.

```
┌─────────────────────────────────────────────────────────────────┐
│ Entry & UI Layer                                                │
│   CLI · TUI · Web (FastAPI/WebSocket) · RPC (stdin/stdout JSONL)│
│   Shared managers: Config, Session, Mode, Approval              │
├─────────────────────────────────────────────────────────────────┤
│ Agent Layer                                                     │
│   Initializer Agent · Coding Agents · Planner subagent          │
│   Generator/Critic pair · Evaluator · Specialist subagents       │
│   Extended ReAct: pre-check → think → critique → act → observe  │
├─────────────────────────────────────────────────────────────────┤
│ Tool & Context Layer                                            │
│   ToolRegistry · MCP lazy discovery · Skills (3-tier hierarchy) │
│   Adaptive context compaction · System reminders · Memory       │
├─────────────────────────────────────────────────────────────────┤
│ Persistence Layer                                               │
│   Session store (tree-structured) · feature_list.json           │
│   harness-progress.txt · git as state machine · Provider cache  │
│   Operation log · Instinct store                                │
└─────────────────────────────────────────────────────────────────┘
         ↕
  Multi-cloud adapter (Azure ACA · GCP Cloud Run · AWS Kiro)
  Durable Functions fan-out/fan-in · Permission classifier
```

The harness is the runtime orchestration that wraps the reasoning loop. Scaffolding is what happens before the first prompt; harness is what happens after.

## 3. The eight closed gaps (with implementation)

### 3.1 Initializer agent + coding agent split

**Problem in v2.0:** All 10 agents had identical prompts modulo role. No agent owns first-context-window setup.

**v3.0:** Introduce a distinct **Initializer Agent** prompt that runs exactly once at project genesis. Responsibilities:

- Expand the user prompt into a comprehensive `feature_list.json` (200+ entries for a non-trivial app), each entry: `{category, description, steps[], passes: false}`.
- Write `init.sh` to start the dev server and run a basic end-to-end smoke test.
- Initialize git repo with `.gitignore`, conventional commits config, and a first commit.
- Create `harness-progress.txt` with project context, architecture decisions, and the empty progress log.
- Write `CLAUDE.md` with project-specific conventions (style, forbidden patterns, test commands).
- Discover and register MCP servers needed for the project (Puppeteer, Playwright, GitHub, project DB).
- Exit cleanly. **Initializer never returns mid-project.**

**Coding agents** (all subsequent sessions) follow a fixed startup sequence enforced by a SessionStart hook:

```
1. pwd
2. read harness-progress.txt
3. read feature_list.json
4. git log --oneline -20
5. run init.sh smoke test
6. select highest-priority failing feature
7. work on exactly one feature
8. on completion: edit feature_list.json (passes: true), git commit, append progress
```

**Implementation:** Two new agent definitions in `agents/`: `initializer.md` and `coding-agent.md` (rename of generic agent). SessionStart hook in `hooks/session-start.js` enforces the startup sequence. Pre-tool-use hook denies edits to `feature_list.json` except for the `passes` field flip.

### 3.2 feature_list.json as the contract

**Problem in v2.0:** Eval ratchet thresholds defined per phase, but no durable artifact survives across sessions to declare "done."

**v3.0:** `feature_list.json` is the source of truth for project completion. Project is complete when all entries `passes: true`. Coding agents:

- Cannot remove or rewrite entries (deny-rule enforced by hook).
- Can only flip `passes: false → true` after end-to-end verification.
- May propose new entries via `/feature-add` command, which spawns a Critic subagent to validate the proposal before merge.

**JSON not Markdown** — the model is less likely to rewrite JSON than prose. Schema validation runs in the PostToolUse hook on every edit.

### 3.3 Ralph Loop: exit interception

**Problem in v2.0:** Coding agent runs context window to exhaustion or declares premature completion. No mechanism to force continuation when the goal isn't met.

**v3.0:** Implement Ralph Loop via Stop-event hook:

```
Stop hook fires →
  read feature_list.json
  if any passes == false:
    if context budget < 30%:
      → compact + reinject original goal + harness-progress.txt summary
      → new context window, same task
    else:
      → reinject "incomplete features remain" reminder
      → continue
  else:
    → run end-to-end suite
    → if all green: terminate
    → else: flip newly-failing features to passes: false, continue
```

The agent cannot exit while features remain failing. This is the missing autonomy mechanism.

### 3.4 Per-workflow LLM routing (compound system)

**Problem in v2.0:** All 10 agents called Claude Opus by default. No cost-aware or capability-aware routing.

**v3.0:** Per-workflow model binding. Each workflow declares model requirements:

| Workflow | Recommended model | Rationale |
|---|---|---|
| Initializer | Opus 4.7 (or current best) | One-shot, high stakes, complex synthesis |
| Planner subagent | Opus 4.7 | Read-only, deep reasoning, plan quality |
| Coding agent (action) | Sonnet 4.6 | Tool-calling workhorse, cost-sensitive |
| Generator (in GAN pair) | Sonnet 4.6 | High-volume code generation |
| Critic (in GAN pair) | Opus 4.7 | Independent quality judgment must be stronger |
| Evaluator | Sonnet 4.6 | Test execution and grading |
| Compaction workflow | Haiku 4.5 | Cheap, fast, summarization-only |
| Self-critique pass | Opus 4.7 | Catches issues the action pass missed |
| Cross-provider failover | OpenAI / Gemini / Bedrock | When Anthropic rate-limited |

Routing config lives in `config/workflows.yaml`. Provider abstraction modeled on Pi: TypeBox schemas for tool definitions, serializable as JSON, supporting Anthropic, OpenAI, Google, Bedrock, Vertex, Azure, Ollama, vLLM endpoints.

### 3.5 Plan Mode as subagent (not state machine)

**Problem in v2.0:** No explicit plan/execute separation. Planning happens implicitly inside the action pass.

**v3.0:** Adopt OPENDEV's pattern. No mode state machine. Instead:

- `/plan` command spawns a **Planner subagent** with a read-only tool schema (no write, no edit, no bash-mutate).
- Planner produces a structured plan file in `scratch/plans/`: goal, context, files to modify, new files, implementation steps, verification criteria, risks.
- Planner returns the plan path to the main agent via `present_plan(plan_file_path)`.
- User approves or requests revision. On approve, main agent proceeds in normal mode.

Schema-level restriction (Planner literally cannot see write tools) eliminates the "stuck in plan mode" failure.

### 3.6 Extended ReAct loop

**Problem in v2.0:** Standard ReAct (reason → act → observe). No explicit thinking or self-critique phase.

**v3.0:** Extended ReAct with six phases per iteration:

```
1. pre-check + compaction       → drain message queue, compact if >70% budget
2. thinking (optional, configurable depth) → chain-of-thought trace
3. self-critique (optional)     → independent pass on the thinking output
4. action                       → LLM call with full tool schema
5. tool execution               → registry dispatch through five safety layers
6. post-processing              → decide iterate vs return
```

Phases 2 and 3 are configurable per workflow (thinking_level: off/min/low/med/high). The self-critique pass uses Opus regardless of action-pass model.

### 3.7 Tool-result budget footer injection

**Problem in v2.0:** Eval ratchet gates at phase boundaries. No per-tool-call budget signal. Agents over-explore.

**v3.0:** Inject budget state into every tool result. Pattern from Stripe Minions and Google BATS research:

```
[tool result body]
---
budget: 12,400 / 200,000 tokens (6.2%) | turn 8 / 40 | regime: NORMAL
```

Regime transitions:

- `NORMAL` (0–60% budget): no special instructions
- `CONSERVE` (60–80%): "prefer reading over searching, prefer targeted edits over rewrites"
- `HIGH` (80–95%): "wrap up current feature, do not start new exploration"
- `CRITICAL` (>95%): "commit current state, write progress note, terminate"

Footer is added by a PostToolUse hook in the tool execution layer. No prompt-engineering; this is mechanically appended.

### 3.8 Mandatory browser-automation end-to-end gate

**Problem in v2.0:** Eval ratchet checks unit tests and lint. Anthropic's research identified this as the #1 silent failure mode — agents declare features done without end-to-end verification.

**v3.0:** Feature `passes: false → true` transition is **blocked at the hook layer** unless:

- A Puppeteer or Playwright MCP session executed the feature's `steps[]`.
- A screenshot or DOM assertion confirmed the expected state.
- The verification artifact is committed to git under `verification/<feature_id>.{png,json}`.

The hook fires on `Edit` operations against `feature_list.json` and rejects the edit if the verification artifact is missing. This is the most important single change in v3.0.

## 4. The eight new components

### 4.1 Five-layer defense-in-depth safety

Adopted from OPENDEV verbatim, mapped onto existing permission classifier:

| Layer | Mechanism | Existing analogue |
|---|---|---|
| 1. Prompt-level guardrails | Security policy, action safety, read-before-edit, git workflow in system prompt | Extends existing CLAUDE.md |
| 2. Schema-level tool gating | Plan-mode subagent has no write tools in schema; per-subagent allowed_tools | New |
| 3. Runtime approval | Manual/semi-auto/auto levels, pattern/command/prefix/danger rules, persistent permissions | Existing permission classifier extended |
| 4. Tool-level validation | DANGEROUS_PATTERNS blocklist, stale-read detection, output truncation, timeouts | New |
| 5. Lifecycle hooks | PreToolUse blocking (exit code 2), argument mutation, JSON stdin | Existing 14 hooks |

Each layer is independent. No single failure compromises the system.

### 4.2 Event-driven system reminders

**Problem:** Long sessions drift. Initial system prompt instructions fade after 50+ turns.

**v3.0:** Targeted reminders injected at decision points, not in the system prompt:

- Before any `Edit` on production code: reminder of project style rules.
- Before any `Bash` invoking `rm`, `dd`, `mkfs`: explicit warning and approval gate.
- Before marking a feature passing: reminder of verification requirements.
- Before subagent spawn: reminder of scope boundaries.
- After 25 turns without a git commit: reminder to checkpoint.

Reminders live in `prompts/reminders/` as small Markdown snippets. The PreToolUse hook decides which to inject based on the tool and arguments.

### 4.3 Adaptive context compaction (five-stage)

**Problem in v2.0:** Context erosion was acknowledged but not actively managed.

**v3.0:** Five-stage progressive compaction running automatically as context fills:

| Stage | Trigger | Action |
|---|---|---|
| 1 | 50% budget | Drop stale tool-call outputs (older than 20 turns) |
| 2 | 65% | Summarize file-read outputs to 200-char abstracts, keep originals on disk |
| 3 | 75% | Summarize older turns into structured notes, keep last 10 turns verbatim |
| 4 | 85% | Aggressive summarization, drop everything except the current feature's work |
| 5 | 92% | Forced checkpoint: git commit, write progress, hand off to Ralph Loop |

Compaction workflow uses Haiku for cost. Original transcripts archived to `sessions/archive/<session_id>/` before summarization. PreCompact hook fires before stages 3-5.

### 4.4 Continuous learning: instinct extraction (ECC v2 pattern)

**Problem in v2.0:** Harness doesn't learn from its own runs.

**v3.0:** Stop hook extracts patterns:

- Parse session transcript for {tool sequence → outcome} tuples.
- Score each by `frequency × success_rate × novelty`.
- Promote high-scoring patterns to **instincts** in `instincts/pending/` with confidence score and example transcripts.
- `/evolve` command clusters related instincts into skills, with the Critic subagent validating before merge to `skills/`.
- `/instinct-export` and `/instinct-import` allow sharing across projects.

Instinct lifecycle: pending → tentative → confirmed → skill. Pending instincts auto-prune after 30 days if not promoted.

### 4.5 Tree-structured sessions (Pi pattern)

**Problem in v2.0:** Linear session history. No ability to branch and try alternatives.

**v3.0:** Sessions stored as trees, not lists:

- `/tree` navigates the session graph in place.
- `/fork` creates a branch from any point.
- `/branch <name>` labels the current path.
- All branches live in a single session file under `sessions/<project>/<session_id>.json`.
- Sessions auto-save per turn, organized by working directory.
- `/export` produces HTML for review.

Branching is essential for hard tasks where the first approach often fails. Lets you try B without losing A.

### 4.6 Three-tier skills hierarchy

**Problem in v2.0:** 23 skills, flat namespace, no override semantics.

**v3.0:** Three-tier skills resolved in priority order:

```
1. project-local:   .harness/skills/        ← overrides everything
2. user-global:     ~/.harness/skills/       ← personal toolkit
3. built-in:        harness_forge/skills/    ← shipped defaults
```

Skill loader merges with project-local winning. Each skill is a directory with `SKILL.md` (description + activation conditions) plus optional `script.sh`, `template.md`, examples. Skills lazy-load via `use_skill <name>` tool, not preloaded into context.

### 4.7 Spec-gap backpropagation (Forge pattern)

**Problem in v2.0:** When eval ratchet fails phase 7, re-run phase 7. But the root cause is often a spec gap at phase 2.

**v3.0:** Failed ratchet triggers a Spec-Audit subagent:

- Reads the failing phase's expected outputs.
- Walks back through phases 6, 5, 4, … reading their specs.
- Identifies the earliest phase whose spec, if tightened, would have prevented the failure.
- Proposes a spec amendment with diff.
- Critic subagent validates the amendment.
- On approve: spec patched, all downstream phases re-run from the amended phase.

This is the missing root-cause loop. Without it, the harness grinds re-running symptoms.

### 4.8 Monotonic-improvement guards (Factory pattern)

**Problem in v2.0:** Eval scores can regress between phases without triggering rollback.

**v3.0:** Every ratchet checkpoint stores a `state_snapshot` (git SHA + feature_list.json + eval scores). Between checkpoints:

- If new eval score >= previous on all dimensions: keep, advance.
- If any dimension regressed: keep-or-revert decision based on user-configurable policy.
- Default policy: revert and try a different approach.
- All keep/revert decisions logged to `experiments/log.jsonl`.

Eval dimensions are auto-discovered (not hardcoded) — the Evaluator workflow inspects the project type and generates appropriate dimensions (test coverage, type-check pass rate, lint score, build success, browser E2E pass rate, performance benchmarks if applicable).

## 6. Cross-cutting concerns

### 6.1 Cost tracking

Every LLM call tagged with `{workflow, model, session_id, feature_id}`. Per-session ledger in `sessions/<id>/cost.json`. `/cost` command shows breakdown. Required for cost-aware routing decisions and for invoicing on consulting deliverables.

### 6.2 Cross-provider failover

Provider abstraction modeled on Pi. On Anthropic rate-limit or outage:

- Critical workflows (Critic, Spec-Auditor) fail over to GPT-5 or Gemini 2.5 Pro.
- High-volume workflows (Coding-Agent action pass) fail over to Bedrock or Vertex.
- Compaction stays on Haiku unless unavailable; falls back to GPT-5-mini.

Mid-session model switch supported; thinking blocks auto-converted to the destination provider's format.

### 6.3 Multi-cloud adapter (unchanged)

Azure ACA Dynamic Sessions, GCP Cloud Run, AWS Kiro paths remain. Durable Functions fan-out/fan-in topology unchanged. Permission classifier extended to be Layer 3 of the new five-layer safety.

### 6.4 Observability

Per-session telemetry:

- Tokens in/out per workflow.
- Tool calls with duration and success.
- Compaction events (which stage, what was dropped).
- Ratchet decisions (keep/revert with diff).
- Instinct candidates extracted.
- Cost ledger.

Telemetry written to `sessions/<id>/telemetry.jsonl`. Optional shipping to Langfuse, Helicone, or Honeycomb via OTel.

### 6.5 Recipes (Goose pattern)

YAML-defined repeatable workflows in `recipes/`:

```yaml
name: weekly-consulting-deliverable
inputs:
  client: string
  week_ending: date
steps:
  - skill: market-research
    params: { client_industry: "engineering-construction" }
  - skill: use-case-prioritization
    params: { dimensions: [feasibility, value, risk, time, cost] }
  - skill: deck-generator
    params: { template: "5-dim-scoring" }
  - skill: doc-updater
    params: { target: "google-slides" }
```

Recipes are deterministic, parameterized, and version-controllable. Run via `/recipe-run weekly-consulting-deliverable.yaml`.

## 7. Phase pipeline (revised from v2.0's 9 phases)

v2.0 had 9 phases. v3.0 retains them but adds gates:

| # | Phase | New gate |
|---|---|---|
| 1 | Intake | Initializer runs, feature_list.json written |
| 2 | Architecture | Critic subagent validates before phase 3 |
| 3 | Plan | Planner subagent (read-only) produces plan; user approves |
| 4 | Generate | Generator with budget footer, compaction stages active |
| 5 | Self-critique | Opus pass independent of generator's model |
| 6 | Test | Unit + integration runs |
| 7 | E2E verify | **Browser automation mandatory before any passes flip** |
| 8 | Code review | Code-Reviewer + Security-Reviewer subagents |
| 9 | Merge | Monotonic-improvement guard; revert if regression on any dimension |

Failure at any phase triggers Spec-Auditor backprop before retry. After 3 retries on same feature: escalate to user via HITL gate.

## 8. Migration path from v2.0

Done in three increments, not a big bang.

**Increment 1 (week 1-2): foundation**

- Add Initializer agent and feature_list.json contract.
- Add E2E gate hook and Puppeteer MCP.
- Add Ralph Loop Stop hook.
- Outcome: existing v2.0 projects retrofitted with the three highest-value Anthropic findings.

**Increment 2 (week 3-4): compound system**

- Add per-workflow model routing config.
- Add Planner subagent + Plan Mode.
- Add Extended ReAct loop (thinking + critique phases).
- Add tool-result budget footer.
- Outcome: cost-aware, plan-driven harness.

**Increment 3 (week 5-6): self-improvement**

- Add Spec-Auditor + backprop loop.
- Add monotonic-improvement guard + experiment log.
- Add instinct extraction + evolve.
- Add tree-structured sessions.
- Add recipe runner.
- Outcome: harness that learns.

After increment 3, run the full retrofit against `SiteMind` and the HCRIS engagement to validate against real workloads.

## 9. Open questions for v3.0

1. **Single coding agent vs specialized fleet?** Anthropic's research left this open. v3.0 starts with single + subagent delegation; revisit after increment 2 data.
1. **Compaction summarization quality.** Haiku is cheap but compaction errors compound. May need a Sonnet-grade evaluator on every stage-3+ compaction.
1. **Instinct promotion threshold.** Need empirical data on `frequency × success × novelty` weights. Start at 0.4 / 0.4 / 0.2; tune from telemetry.
1. **Cross-provider thinking-block conversion.** Pi handles this; need to verify accuracy on Anthropic → OpenAI → Anthropic round-trips.
1. **Multi-tenant isolation.** Consulting use case will have multiple clients on the same harness instance. Per-client session/skill/instinct namespacing TBD.

## 10. Repo harvest map — what to vendor instead of generate

**Principle.** Every line of code in v3.0 that already exists in a permissively-licensed public repo should be vendored, not generated. Generating "from understanding" of the source paper burns tokens, drifts from the reference, and skips bug fixes the maintainers have already shipped. The table below names the exact repo, the path within it, the license, and the integration mode for every v3.0 component that has a public reference.

**Integration modes:**

- **Vendor** — copy files into `harness_forge/vendor/<repo>/` with a `LICENSE` and `UPSTREAM.md` noting commit SHA and last-sync date. Apply patches in `harness_forge/patches/<repo>.patch`.
- **Fork** — full hard fork under our org, maintain divergence. Use when we'll diverge significantly.
- **Depend** — import as a package (npm/pip). Use when the upstream is well-versioned and stable.
- **Read-and-reimplement** — license incompatible, architecture conflicts, or domain-specific. Read the source as a spec; write our own. Should be the last resort.

### 10.6 Vendoring discipline

For every vendored module, the following are mandatory:

1. `vendor/<repo>/UPSTREAM.md` containing:
   - Source URL
   - Commit SHA at time of vendor
   - License text (copied)
   - List of files vendored
   - List of local patches applied
   - Last-sync date and last-reviewed date
1. `vendor/<repo>/LICENSE` — verbatim from upstream.
1. Patches isolated in `patches/<repo>.patch` (not inlined into vendored files), so re-syncing upstream is mechanical.
1. CI job that flags vendored modules unreviewed for >90 days. Forces a manual confirm-or-update.
1. Attribution in `NOTICE.md` at repo root.

### 10.7 Estimated reuse leverage

Rough estimate of v3.0 implementation effort with vs. without the reuse discipline:

| Approach | Token cost (LLM-generated) | Engineering hours | Drift risk |
|---|---|---|---|
| Generate everything from spec | ~2.5M tokens across 6 weeks | ~280 hours | High — diverges from reference implementations |
| Vendor + depend per §10 map | ~0.6M tokens (mostly glue and patches) | ~110 hours | Low — sync upstream periodically |
| Hybrid (vendor anchors, generate glue) | ~1.1M tokens | ~160 hours | Medium |

**Recommendation: full vendor mode for increments 1 and 2.** Reserve generation for the §4.7 spec backprop and §4.8 monotonic guard, where the public references are thinner.
