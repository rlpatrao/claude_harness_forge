## v3.2.1 — learned-rules.md propagation

### Why

Their `learned-rules.md` is a session-persistent list of "rules the human distilled after review" that gets **injected verbatim** into every skill's context-gathering phase — `/change`, `/vibe`, `/feature` in their harness. Cheap, universal reinforcement of hard-won lessons.

We have `instincts/pending/tentative/confirmed` (v3.0 §4.4) as a *promoted* rule store — Critic-gated, cross-project. It's a different scope: derived / distilled. The learned-rules.md pattern fills the gap between "raw human note" and "promoted instinct" — a fast lane that doesn't require Critic review.

### What we build

- **Contract:** `state/learned-rules.md` — one rule per bullet, human-editable, session-persistent. Seed as empty stub.
- **Injection:** three touchpoints
  - `hooks/session-start.js` reads it and appends a "Learned rules (v3.2.1)" block to the SessionStart reminder
  - `agents/coding-agent.md` gains an explicit "Read `.claude/state/learned-rules.md` before selecting a feature; treat every non-empty bullet as a hard preference"
  - `agents/planner.md`, `agents/critic.md`, `agents/spec-writer.md`, `agents/generator.md` gain the same read-and-inject line in their Phase 1 (Read Context)
- **Cross-lane touch:** `skills/build/SKILL.md`, `skills/implement/SKILL.md`, `skills/refactor/SKILL.md`, `skills/change/SKILL.md` gain the read-and-inject step
- **Cap:** 16KB total across all injections (SessionStart truncates past that)
- **Not built:** an `/evolve`-style promotion from `state/learned-rules.md` to `instincts/confirmed/` (deferred; this is meant to be a fast lane)

### Verification

- Seed `state/learned-rules.md` with 3 rules
- SessionStart reminder contains "Learned rules (v3.2.1)" block with all 3 rules
- 16KB cap enforced (drops entries past cap with a marker)
- Coding-agent, planner, critic frontmatter each carries an explicit "read learned-rules" step

## v3.2.2 — 3-instance majority vote

### Why

BRD §3.6 already runs a Critic pair on generator output. The external harness's Devin-parity item 1 adds a **3-fresh-context spawn** at merge boundary (their `/gate`) — 2-of-3 per axis, non-returning voter fails safe to BLOCK. Real defense against single-verifier variance, at 3× cost per merge (not per iteration).

For us, the analog merge boundary is the `feature_list.json` `passes:false→true` flip enforced by [`hooks/e2e-gate.js`](../hooks/e2e-gate.js). That's where we'd want a majority vote.

### What we build

- **New command:** `commands/critic-vote.md` — `/critic-vote <feature-id>`. Wraps a skill invocation.
- **New skill:** `skills/critic-vote/SKILL.md` documents the flow.
- **New script:** `scripts/critic-vote.js` — orchestrator. Spawns 3 fresh `critic` invocations via the SDK Agent tool (fallback: sequential-but-context-independent), collects verdicts, computes majority per axis, writes `verification/<feature_id>.votes.json`.
- **Hook wiring:** [`hooks/e2e-gate.js`](../hooks/e2e-gate.js) extended: on `passes:false→true` flip, require `verification/<id>.votes.json` exists with `verdict: "APPROVED"` and `majority_axes >= 2` (out of N axes).
- **Scope:** ONLY at final feature merge (E2E gate). NOT at per-turn Critic invocations in the auto-loop (would triple cost of every iteration).
- **Failure mode:** non-returning voter counted as BLOCK for that axis (fail safe). If majority BLOCKs any axis, the flip is rejected.

### Verification

- Vote script produces a valid votes JSON with schema
- e2e-gate reads the votes JSON and rejects flip if majority BLOCKs
- Skipping the vote (no votes.json) still rejects the flip
- Verified against a fake feature entry

## v3.2.3 — Sensor arbitration + waivers

### Why

Our forge has 27 hooks. Some (feature-edit-guard, e2e-gate, detect-secrets) hard-block. Others (lint-on-save, typecheck) self-correct in the model. Others (token-advisor, reminder-injector) only advise. There's no cross-cutting taxonomy or waiver mechanism — an override to a specific hook has to be a code edit.

The external harness's `docs/sensor-arbitration.md` + `sensor-waivers.json` schema formalizes this. Adopt.

### What we build

- **New doc:** `docs/sensor-arbitration.md` — the 4-level taxonomy:
  - `hard-block` (exit 2 with BLOCKED message; requires waiver to bypass)
  - `self-correct` (agent gets the error, is expected to fix)
  - `review-focus` (advisory to the reviewer at merge boundary)
  - `advisory` (informational only)
- **Classify all 27 hooks** with their current level in the doc
- **New schema:** `templates/sensor-waivers.schema.json`
- **New helper:** `scripts/check-waiver.js` — helper each hook can call: `checkWaiver(hookName, subject, cwd) → { waived: bool, reason?: string, expires_at?: string }`
- **Wired reference:** [`hooks/pre-bash-gate.js`](../hooks/pre-bash-gate.js) consults check-waiver.js before blocking; other 26 hooks documented but not wired in this increment (per-hook wiring is mechanical; the taxonomy + helper are the leveraged part)
- **Waiver file:** `specs/reviews/sensor-waivers.json` — array of `{ sensor, subject, reason, granted_by, granted_at, expires_at }`

### Verification

- Waiver schema validates a sample waiver
- check-waiver.js returns waived:true for a matching current entry
- check-waiver.js returns waived:false for expired entries
- pre-bash-gate hooked test: with an active waiver for `.env` write, the write is allowed with an audit log line

## v3.2.4 — Cross-feature regression sensor

### Why

Our BRD §3.8 E2E gate re-verifies **only the feature being flipped**. G15 is exactly the failure mode we haven't defended: an edit that passes its own steps[] but silently breaks an earlier passing feature's steps[].

### What we build

- **New script:** `scripts/impact-scope.js` — computes changed files (via `git diff --name-only`) → reverse-dep blast radius over `state/code-graph.json` → owning feature IDs. Uses import edges + file-path prefix matching.
- **New script:** `scripts/regression-gate.js` — full sweep. Reruns every previously-`passes:true` feature's steps[] via the same Playwright/Puppeteer MCP path the E2E runner uses. Emits `verification/regression-<git-sha>.json` summary. Slow — run at merge only.
- **New script:** `scripts/local-regression-gate.js` — impact-scoped variant. Runs only features whose owning-files are in the impact set. Fast — safe to run per-feature.
- **Hook wiring:** [`hooks/e2e-gate.js`](../hooks/e2e-gate.js) extended: on flip, require `verification/regression-<git-sha>.json` OR `verification/<id>.local-regression.json` (whichever is present and current). Full-sweep is the safe default; local variant is the fast path.
- **Scope caveat:** Only re-runs deterministic Playwright/Puppeteer steps captured in each feature's `steps[]`. Non-E2E-covered behavior is out of scope for this sensor.

### Verification

- impact-scope.js produces a blast radius for a synthetic diff
- regression-gate.js runs against the seeded verification artifacts and reports pass/fail per feature
- local-regression-gate.js only runs impacted features
- e2e-gate rejects flip when regression-<sha>.json is missing

## v3.2.5 — Balanced Coupling rubric

### Why

Our `critic` and `code-reviewer` currently reason about coupling in prose. The external harness's G27 grounds this in Khononov's 3-axis Balanced Coupling with a formal rule:

```
BALANCE = (STRENGTH XOR DISTANCE) OR NOT VOLATILITY
```

Cheap, prompt-only, published — no reason not to adopt.

### What we build

- **Update:** [`agents/critic.md`](../agents/critic.md) — add "Balanced Coupling" section with the rule + a per-file worksheet (STRENGTH: intimate/weak; DISTANCE: same-file/module/service; VOLATILITY: high/low). Include 2 worked examples.
- **Update:** [`agents/code-reviewer.md`](../agents/code-reviewer.md) — same.
- **Update:** [`skills/architect/SKILL.md`](../skills/architect/SKILL.md) — reference the rule in the review-doc template (v3.1.2 review loop).
- **New reference doc:** `skills/critic/references/balanced-coupling.md` — the extended writeup with worked examples.

### Verification

- Grep confirms all three files reference the rule
- Reference doc renders
