## 1. Problem / motivation

The forge learns preferences three ways today, and **all of them are advisory** — they rely on an LLM (Critic, generator, planner) *choosing* to honor a rule that sits in its context:

| Existing lane | Store | How it's applied |
|---|---|---|
| Human fast-lane rules (v3.2.1) | `state/learned-rules.md` | Injected verbatim at SessionStart; Critic "applies as hard filters" |
| Mined instincts (v3.0 §4.4) | `instincts/{pending,tentative,confirmed}/` | Clustered into advisory **skills** via `/evolve` |
| Cross-project learnings | `learnings/` | Read as reference |

None of them **block a violating action before it happens.** A rule like "never hardcode secrets" or "always null-check the session object" is only caught *after* the fact, at a gate or by the Critic — costing a full generate→reject→regenerate cycle. This is the "access-compliance gap": remembering a correction ≠ enforcing it.

**TRACE closes the loop with two additions the forge lacks:**

1. **Acquisition** — mine rule candidates automatically from the correction signals the forge *already produces* (Critic BLOCK verdicts, failed ratchet gates, e2e-gate rejections). No human typing required.
2. **Compiled enforcement** — a deterministic PreToolUse gate (`rule-gate.js`) that evaluates compiled rules against a pending Edit/Write/Bash and **blocks** it before the model can commit the violation. This shifts the ratchet *left*: the cheapest place to stop a known-bad pattern is before it's written.

### Design decision (locked): correction source = **A**
Mine corrections from **Critic rejections + failed gates** (not human chat). Rationale: the forge's primary mode is autonomous with no human in the loop, so Critic/gate rejections are the densest, already-structured correction stream. The detector is written as an **adapter over correction sources**, so a human-chat adapter (option C) is a purely additive later change — no rework.

## 2. Architecture

Two rule stores, split by *how a rule is enforced* rather than by origin:

```
state/learned-rules.md        (EXISTING, unchanged)  — human fast-lane, advisory, injected
state/compiled-rules.json     (NEW)                  — machine rules with a check spec + lifecycle
state/rule-candidates/*.json  (NEW)                  — mined, un-promoted (mirrors instincts/pending)
state/rejections.jsonl        (NEW)                  — append-only normalized correction signal
```

Each compiled rule carries a **check spec** that decides *where* it is enforced:

- `check.kind: "pattern"` → **deterministic**. A regex / path-glob / forbidden-substring test that `rule-gate.js` runs **inline in the PreToolUse hook** and can hard-block. (e.g. "no `AKIA[0-9A-Z]{16}` in source", "no `.only(` in test files", "no `console.log` in `src/`".)
- `check.kind: "semantic"` → **LLM-judged**. Cannot be expressed as a pattern (e.g. "don't widen a public API without updating its docstring"). NOT run in the hook (keeps the hook synchronous and cheap). Instead it is surfaced to the **Critic** as an extra hard-filter bullet, reusing the existing `learned-rules.md` injection path. Deterministic rules block early; semantic rules block at the Critic gate — strictly better than today, where semantic rules also only reach the Critic but are hand-authored.

This split is the crux: **cheap, unambiguous rules become early hard blocks; fuzzy rules stay Critic-enforced but are now auto-acquired.**

### Lifecycle (mirrors the instinct lifecycle deliberately)

```
candidate  -- mined by correction-detector.js from a repeated rejection; lives in state/rule-candidates/
tentative  -- /rules promoted after Critic validation; enforced as WARN (non-blocking) in rule-gate.js
confirmed  -- recurred across >= N sessions (default 2) with no false-positive override; enforced as BLOCK
retired    -- superseded/duplicate; kept in compiled-rules.json with status:"retired" (monotonic, never deleted)
```

Severity escalates with confidence: **tentative → warn, confirmed → block.** This honors the forge's "nothing hardens into hard enforcement without Critic + recurrence" ethos (same guarantee `/evolve` gives skills).

## 3. Components to build

### 3.1 `state/compiled-rules.json` (NEW — seed stub)
```json
{
  "version": "3.3.0",
  "comment": "Monotonic. Rules are never deleted; superseded rules get status:'retired'. Produced by /rules from state/rule-candidates/. Consumed by hooks/rule-gate.js (pattern rules) and agents/critic.md (semantic rules).",
  "rules": []
}
```
Rule object schema:
```json
{
  "rule_id": "sha1-12",
  "statement": "Never hardcode AWS access keys in source files.",
  "why": "Recurred as a security-review BLOCK in 3 sessions.",
  "applies_when": { "tools": ["Edit","Write"], "path_glob": "src/**" },
  "check": { "kind": "pattern", "test": "regex", "value": "AKIA[0-9A-Z]{16}", "flags": "" },
  "status": "confirmed",
  "severity": "block",
  "origin": { "source": "security-review", "sessions_seen": 3, "candidate_id": "..." },
  "created_at": "ISO", "confirmed_at": "ISO",
  "false_positive_overrides": 0
}
```
For `check.kind:"semantic"`, `check` is `{ "kind": "semantic", "value": "<natural-language test the Critic evaluates>" }` and `severity` is always `critic` (never `block` — semantic rules can't hard-block in the hook).

### 3.2 `state/rejections.jsonl` (NEW — normalized correction signal, append-only)
One JSON object per line. The **producers** append via a shared helper (see 3.7):
```json
{"ts":"ISO","session_id":"...","source":"critic|security-review|e2e-gate|feature-edit-guard|ratchet","verdict":"block","reason":"hardcoded secret in src/auth.ts","file":"src/auth.ts","tool":"Write","excerpt":"const KEY='AKIA...'"}
```

### 3.3 `hooks/correction-detector.js` (NEW — Stop hook)
- Registered on `Stop`, **after** `instinct-extractor.js` (so it runs in the same curation pass).
- Reads `state/rejections.jsonl` for this session + prior sessions.
- Groups by normalized `reason`/`source`; a group seen **>= 2 times** becomes a candidate written to `state/rule-candidates/<hash>.json`.
- Candidate includes a **proposed** `check` spec: the detector attempts to synthesize a `pattern` (e.g. lift the offending substring/regex from `excerpt`); if it can't, it marks the candidate `check.kind:"semantic"` with the reason text as the NL test.
- **Defensive:** exit 0 on any error. Never writes to `compiled-rules.json` or `skills/` — only to `state/rule-candidates/`. (Same guarantee as `instinct-extractor.js`.)
- Dedup via `state/.rule-candidate-hashes.txt` (mirror of `instincts/.seen-hashes.txt`).

### 3.4 `scripts/rule-compile.js` + `commands/rules.md` (NEW — the `/rules` curation command)
Mirror of `/evolve` / `scripts/instinct-evolve.js`. Subcommands:
```
node scripts/rule-compile.js status              # counts by lifecycle stage
node scripts/rule-compile.js promote-candidates  # candidate → tentative (AFTER Critic validation per candidate)
node scripts/rule-compile.js promote-tentative   # sessions_seen>=2 & 0 false-positive overrides → confirmed (severity block)
node scripts/rule-compile.js compile             # write validated rules into state/compiled-rules.json
node scripts/rule-compile.js prune-candidates    # delete candidates older than 30 days
```
Hard rules (copy the `/evolve` guarantees):
- **No auto-promotion past `candidate` without Critic.** The Critic must confirm the candidate describes a real, non-coincidental preference AND that its `pattern` check has no obvious false positives (Critic sees sample matches). `--force` bypass exists but is discouraged.
- **candidate → tentative only via Critic; tentative → confirmed only via recurrence** (>= N sessions, default 2).
- **The user reviews before a rule reaches `block` severity.** `/rules compile` stages the diff to `compiled-rules.json`; user commits after review. (Matches "no auto-merge to skills/".)

### 3.5 `hooks/rule-gate.js` (NEW — PreToolUse hook, the enforcement half)
- Registered on `PreToolUse` for `Edit|Write` and (separately) `Bash`.
- Loads `state/compiled-rules.json`; for each rule where `check.kind==="pattern"` and `applies_when` matches the pending tool + target path:
  - Evaluate the pattern against the **new content** (Edit `new_string` / Write `content` / Bash `command`).
  - On match: if `severity==="block"` → emit `{ "decision": "block", "reason": "<rule.statement> (learned rule <rule_id>)" }` (exit 2). If `severity==="warn"` (tentative) → emit an advisory `reminder` note, exit 0.
- **False-positive escape hatch:** if the model/user overrides a warn, `rule-gate.js` (or a companion PostToolUse) increments `false_positive_overrides` on that rule; a rule with >0 overrides cannot auto-promote to `confirmed`/`block` — it goes back to the Critic. This prevents a noisy pattern from hardening into a block.
- **Defensive:** malformed rules are skipped, never crash the hook; exit 0 on any internal error (fail-open, like the other PreToolUse gates).

### 3.6 `skills/compiled-rules/SKILL.md` (NEW — reference skill)
Frontmatter (`name`, `description`, `when_to_use`, `brd_ref: §v3.3`) + body documenting: the two rule stores and when to use each, the lifecycle, the pattern-vs-semantic split, how to hand-author a compiled rule, and the false-positive override contract. Mirror the shape of `skills/instinct-extraction/SKILL.md`.

### 3.7 Producer wiring (small edits to existing components)
Add a tiny shared helper `hooks/lib/log-rejection.js` (`appendRejection(obj)` → append line to `state/rejections.jsonl`, defensive). Wire the existing rejection points to call it:
- `agents/critic.md` — instruction: on a BLOCK / needs-revision verdict, append a rejection record (source `critic`).
- `hooks/e2e-gate.js`, `hooks/feature-edit-guard.js` — on rejection, append a record.
- Ratchet gate failures (security-review, code-reviewer, mutation, etc.) — append via the gate runner. (If a single gate runner doesn't exist, wire the highest-signal two — `security-review` and `code-reviewer` — in v3.3 and defer the rest.)
- `agents/critic.md` — extend the existing "Learned rules" Read-Context step to ALSO read `check.kind:"semantic"` rules from `compiled-rules.json` and apply each as a hard filter (same treatment it already gives `learned-rules.md`).

## 4. Build order (each step independently testable)

1. **Schemas + stubs** — `state/compiled-rules.json`, `state/rule-candidates/.gitkeep`, `hooks/lib/log-rejection.js`. Seed `compiled-rules.json` empty.
2. **Enforcement first** — `hooks/rule-gate.js` + register in `settings.json`. Hand-write ONE pattern rule (the AWS-key example) directly into `compiled-rules.json` with `severity:"block"` and prove a `Write` of `const K="AKIA...":16` is blocked, and an innocent write passes. *(Enforcement is the highest-value half; ship it working against a hand-authored rule before automating acquisition.)*
3. **Acquisition** — `hooks/correction-detector.js` + `hooks/lib/log-rejection.js` producers. Simulate a `rejections.jsonl` with a repeated security-review block; prove a candidate lands in `state/rule-candidates/`.
4. **Curation** — `scripts/rule-compile.js` + `commands/rules.md`; wire Critic validation. Prove candidate → tentative(warn) → confirmed(block) with recurrence, and that `/rules compile` stages a reviewable diff.
5. **Semantic path** — extend `agents/critic.md` to consume semantic compiled rules; add one semantic rule and prove the Critic applies it.
6. **Docs** — `skills/compiled-rules/SKILL.md`; update root `CLAUDE.md` component table and `feature_list.json` (append-only; via `/feature-add`, respecting `feature-edit-guard`).

## 5. Verification (gate 12 / smoke-launch discipline — use real data, not synthetic fixtures)
- **Enforcement:** a real Write of a file containing a live-looking AWS key pattern is blocked with the rule's reason; a benign file is untouched. Confirm `rule-gate.js` exits 0 (fail-open) when `compiled-rules.json` is malformed.
- **Acquisition:** run an actual session that triggers a security-review BLOCK twice; confirm a candidate with a synthesized `pattern` appears.
- **No regressions:** all existing PreToolUse hooks still fire; `settings.json` still parses; Ralph Loop + instinct-extractor still run on Stop.
- **False-positive safety:** a `warn` rule that gets overridden does NOT auto-promote to `block`.
- Follow the forge's own gates: unit tests for `rule-gate.js` matcher logic + `correction-detector.js` grouping; lint/types clean.

## 6. Non-goals (YAGNI for v3.3)
- Human-chat correction adapter (option C) — deferred; detector is already adapter-shaped.
- Running semantic rules inline in the PreToolUse hook (would make the hook an LLM call — rejected on latency/cost). Semantic rules stay Critic-enforced.
- Cross-project compiled-rule sharing — later, via the existing `/instinct-export`/`/instinct-import` pattern.
- AST-grep / structural checks — v3.3 patterns are regex/glob/substring only; structural checks are a later `check.kind`.
